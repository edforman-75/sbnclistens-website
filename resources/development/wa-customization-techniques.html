1. Purpose and Audience

This page is written for SBNC volunteers, designers, and technical helpers who are deciding how far to push Wild Apricot (WA) customization. It does not tell us what we must do; instead it lays out the tools, trade-offs, and risks for each level of customization so we can make informed decisions.

A key goal is site stability and uptime. Wherever possible, we prefer solutions that:

  • Survive Wild Apricot upgrades with minimal breakage
  • Work consistently across major browsers and devices
  • Are maintainable by future SBNC volunteers

2. Key Criteria for Comparing Approaches

When we evaluate any customization, we should consider:

  • Complexity of implementation
    • How difficult is it to build the first version?
    • Does it require CSS, JavaScript, or API expertise?
  • Ongoing maintenance cost
    • Who can realistically maintain this over the next 3–5 years?
    • Will small content changes require technical help?
  • Cross-browser and device compatibility
    • Does it work on current versions of Safari, Chrome, Edge, Firefox?
    • Is it responsive on desktop, tablet, and mobile without per-device tweaks?
    • Does it behave well for both mouse/trackpad and touch interactions?
  • Dependence on Wild Apricot internals
    • Does it rely on WA HTML structure or class names that may change?
    • Will a theme switch or WA update likely break the customization?
  • Reliability and uptime
    • Does this introduce new points of failure (external APIs, automation tools)?
    • What happens if the external service is down or rate-limited?

The rest of this page walks through the main techniques in roughly the order we would consider them, from simplest and most stable to most flexible (but also more fragile).

3. Staying Inside Wild Apricot Themes and Widgets

3.1 Built-In Themes and Layout Options

Every WA site starts with a theme. SBNC will likely choose a modern responsive theme and stick with it for stability. Themes provide:

  • Header, menu, and footer layout
  • Default typography and color palette
  • Responsive behavior for common layouts
  • Standard behavior for system pages (login, profile, events, etc.)

Pros:

  • Least fragile: changes are supported by WA’s own UI
  • Responsiveness and browser compatibility are handled by the theme
  • No code to maintain; most changes are point-and-click

Cons:

  • Design freedom is constrained to what the theme allows
  • Certain layout ideas (magazine-style home pages, unusual navigation patterns) may not be possible
  • Inconsistent behavior if we mix legacy and modern themes

See the Theme Showcase (separate document) for visual examples and notes on the strengths and weaknesses of individual themes.

3.2 Built-In Wild Apricot Widgets

WA widgets (event lists, member lists, login boxes, etc.) are the building blocks of most pages. They are theme-aware and updated by WA when the platform evolves.

Pros:

  • Safest choice for data that lives in WA (events, members, registrations)
  • Benefits from WA performance and access control
  • Minimal compatibility risk across browsers and devices

Cons:

  • Limited control over layout and visual design
  • Filtering, search, and grouping options can be constrained
  • Sometimes “just short of” what we want (for example, event calendar behavior)

4. Using Custom Fields

Custom fields let us extend WA’s data model (for example, adding “Interest Group” to member profiles or “Event Category” to events).

Typical uses:

  • Tagging events for filtering (Hikes, Wine, Newbies, Public vs Member)
  • Storing consent or preferences (photo permissions, communication preferences)
  • Capturing structured data from event registrations

Pros:

  • Clean way to represent SBNC-specific concepts within WA
  • Works with WA exports, saved searches, and some widgets
  • Stable and supported by the platform

Cons:

  • More fields can make admin screens cluttered
  • Some widgets do not expose custom fields directly
  • Complex logic may require API or automation tools on top of custom fields

In practice, SBNC should treat custom fields as a first-line tool for tagging and segmentation before resorting to external databases.

5. Custom CSS: Styling Without Changing Structure

Custom CSS allows us to change colors, spacing, fonts, and some layout details while keeping the underlying WA HTML structure intact.

Typical uses:

  • Aligning fonts and colors with SBNC brand
  • Improving spacing, readability, and button styles
  • Highlighting key actions (Join, Renew, RSVP, etc.)

Pros:

  • Powerful way to improve visual design without touching data or logic
  • Usually stable across WA updates, as long as we avoid targeting brittle selectors
  • Small CSS changes can have large impact on perceived polish

Cons:

  • Requires CSS skills and disciplined naming
  • Too much CSS can become hard to debug for later volunteers
  • Over-aggressive overrides may break responsiveness on some devices

For SBNC, custom CSS is likely the main tool for visual improvement. It should be documented, commented, and kept in a single central file so future admins can understand what is happening.

6. Custom HTML Gadgets

WA’s Custom HTML gadgets let us embed arbitrary HTML, CSS, and JavaScript into a page. This is how we would embed external calendars, photo galleries, maps, and chat widgets.

Typical uses:

  • Embedding third-party widgets (NewPath CalendarPlus, Tockify, Jotform, Elfsight, etc.)
  • Adding“microsites” or special layouts that WA widgets cannot express
  • Running small JavaScript helpers (for example, conditional content on a page)

Pros:

  • Very flexible; we can implement almost any front-end behavior here
  • Good way to integrate specialized tools without leaving WA entirely
  • Can be isolated to specific pages so risks are contained

Cons:

  • Highest risk for cross-browser issues if not carefully tested
  • External widgets may change or deprecate APIs over time
  • JavaScript errors in a gadget can degrade page performance or break features

For SBNC, Custom HTML gadgets should be used for well-chosen external tools (calendars, photo galleries, maps) where the benefit is clear and the vendor maintains good cross-browser support.

7. External Widgets and Embeds

The Widget Gallery lists specific tools we are considering. This section focuses on the general pattern.

Advantages of external widgets:

  • Richer feature sets (filtering, search, visual layouts, animations)
  • Vendors often handle responsiveness and cross-browser quirks for us
  • We can use the same widget across multiple sites if needed

Risks and trade-offs:

  • We depend on the vendor’s uptime and pricing model
  • Embedding via iframes may have subtle scrolling or sizing issues on mobile
  • Accessibility and keyboard navigation quality varies by vendor

Our Calendar & Photos Strategy document analyzes specific options in more detail, including iframe behavior across browsers.

8. WA API, Webhooks, and Automation Tools (Make / Zapier)

When point-and-click configuration is not enough, WA exposes an API and webhooks that can be combined with automation platforms such as Make or Zapier.

Typical uses:

  • Synchronizing WA data with external tools (mailing lists, analytics, dashboards)
  • Triggering workflows when members join, renew, or register for events
  • Pre-processing data for use in external widgets

Pros:

  • Allows automation without writing and hosting our own server-side code
  • Powerful for connecting WA to other SaaS tools SBNC may use
  • Workflows can often be inspected and edited via a visual interface

Cons:

  • Introduces a new point of failure (the automation platform itself)
  • APIs and webhooks require careful security and rate-limit handling
  • Complex scenarios can become hard to understand for future admins

For SBNC, automation should be used sparingly and intentionally where it clearly saves recurring manual work or enables features WA cannot provide on its own.

9. Full Custom Front-Ends (What We Are Not Doing)

It is technically possible to treat WA purely as a membership back-end and build a separate front-end in WordPress, Webflow, or a custom framework. This would give maximum design freedom but at the cost of complexity and risk.

Why this is not our path for SBNC:

  • Requires significant development and DevOps effort
  • Two sources of truth for authentication, permissions, and content
  • Higher risk of outages and hard-to-debug integration failures

Instead, SBNC’s strategy is to stay primarily inside WA, using themes, custom fields, custom CSS, carefully selected widgets, and light automation to achieve our goals.

10. How This Fits With Platform Limitations

Some constraints in WA apply no matter how clever our customization is. These are summarized in the companion document Wild Apricot Platform Limitations.

Designers should assume that the overall page structure and certain system behaviors are fixed. Customization Techniques describes how far we can go within those boundaries; Platform Limitations describes the boundaries themselves.

11. Summary for the Website Committee

The rough “ladder” of customization options is:

  1. Pick a solid modern WA theme and use built-in widgets
  2. Add custom fields to capture SBNC-specific data
  3. Use custom CSS for visual polish and brand alignment
  4. Embed carefully chosen external widgets via Custom HTML gadgets
  5. Use WA API, webhooks, and automation tools selectively for high-value workflows

The Website Committee can treat this as a menu of tools, not a mandate. For each new feature, we should ask:

  • Can we do this cleanly with themes, widgets, and custom fields alone?
  • If not, does custom CSS or an external widget solve the problem without hurting stability?
  • Only if needed, should we introduce API or automation complexity.
