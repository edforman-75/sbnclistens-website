1. Purpose and Audience

This page explains how the main Wild Apricot (WA) data objects relate to each other, which objects are available via the WA API, and how SBNC extends this model to represent committees and leadership roles.

It is intended for:

  • Designers and developers working on SBNC’s WA customizations.
  • Volunteers configuring widgets, pages, and automation (API, Make-style tools).
  • Future admins who need to understand where committee and role data actually lives.

The goal is not to cover every WA entity, but to focus on the objects SBNC actually uses and extends.

2. Core Wild Apricot Objects We Rely On

At SBNC, most of our workflows center around a few key WA object types. Below is a simplified list, with notes about how we use them and whether they are exposed via the official WA API.

2.1 Contacts & Members

  • Contacts are the base records for people in the database (email, name, address, etc.).
  • Members are contacts who currently hold a membership level.
  • SBNC uses custom fields on contacts/members heavily to track:
    • Leadership roles (e.g., President, VP Activities, Committee Chair).
    • Committee affiliations and responsibilities.
    • Internal flags that drive permissions and internal workflows.
  • API: Contacts and Members are available via the WA API, including custom fields (subject to API permissions).

2.2 Membership Levels

  • Membership levels define the type of relationship a member has with SBNC (e.g., Newcomer, Extended Newcomer, Admin, etc.).
  • SBNC sometimes uses membership levels as role buckets, where a specific level implies broader privileges or responsibilities.
  • API: Membership levels are readable via the WA API and can be used to filter members and drive automations.

2.3 Events & Registrations

  • Events are used both for actual SBNC activities and, in some cases, as pseudo-events that encode internal structures (e.g., committees).
  • Registrations represent people signed up for an event.
  • SBNC uses registrations on pseudo-events to represent committee membership (more on this below).
  • API: Events and Registrations are available via the API, including event categories, registration types, and custom fields.

2.4 Financial Objects (Invoices, Payments, Donations)

  • WA tracks invoices, payments, and donations, usually tied to either memberships or events.
  • SBNC primarily uses these for standard membership and event payments, not for committee modeling.
  • API: Financial objects are partially exposed via the API (details depend on WA plan and permissions).

2.5 Website Content (Pages, Menus, Gadgets)

  • Public and member-facing pages, navigation, and gadgets form the surface layer.
  • From a data-model perspective, pages and gadgets display data from the objects above; they do not store core member or committee data themselves.
  • API: Page content is not exposed as structured objects through the standard WA API. Custom integrations are usually built around pages, not through them.

  • Event Categories group events for calendar views and filtering.

SBNC’s committee and role modeling builds on top of this structure rather than introducing new native objects.

4. API Accessibility and Why It Matters

For SBNC’s automation (including potential use of the WA API, Make, or similar tools), it is important to know which objects can be queried programmatically.

4.1 High-level API view

  • Contacts/Members: API-readable and updatable, including custom fields.
  • Membership Levels: API-readable; can be used for filtering and logic.
  • Events & Registrations: API-readable; core for automation around attendance and committee pseudo-events.
  • Financial Records (Invoices/Payments/Donations): Partially API-readable; used mainly for reporting.
  • Pages, menus, and widgets: Not first-class API objects; usually managed via the admin UI.

4.2 Practical implications for SBNC

  • Anything we want to automate (e.g., leadership rosters, committee lists, special emails) should be represented through API-visible objects like:
    • Custom contact/member fields
    • Events and registrations
  • Design decisions that rely heavily on page structure or theme markup are harder to automate, because those are not API-first.
  • For new customizations, we get more long-term leverage by modeling data in contacts/members and events, with pages acting as views on that data.

5. SBNC Extensions: Roles and Committees

Wild Apricot does not have native objects for “Committee” or “Leadership Role.” SBNC models these concepts using a combination of membership levels, custom fields, and special events.

5.1 Roles via Membership Levels and Custom Fields

  • Some roles are represented as membership levels (for example, admin-level memberships that imply broader access).
  • More granular roles (e.g., “VP Activities,” “Event Chair,” “Tech Committee Member”) are modeled via custom fields on the contact/member record.
  • These custom fields answer questions such as:
    • What leadership role(s) does this member hold?
    • Which committee(s) is this member part of?
    • What is their primary responsibility?
  • Because these are contact/member fields, they are API-accessible, which allows automated exports, dashboards, and roster pages.

5.2 Committees via Pseudo-Events and Registrations

  • Each committee is represented by a special event (sometimes called a pseudo-event).
  • The members of a committee are stored as registrants on that pseudo-event.
  • This lets SBNC reuse WA’s built-in registration UI and API to:
    • Maintain committee rosters.
    • Track join/leaving dates via registration history.
    • Drive automated communications based on committee membership.
  • Important: These pseudo-events must be kept out of public calendars and event listings. At the moment, SBNC needs to double-check how this is configured (categories, visibility settings, or naming conventions) to ensure committee events do not appear inadvertently.

5.3 Combined picture

Putting this together, an SBNC committee member might be represented as:

  • A Contact/Member with:
    • A standard SBNC membership level (e.g., Newcomer, Extended Newcomer).
    • Custom fields indicating current role(s) and committee(s).
  • One or more registrations on committee pseudo-events, providing a roster and history for that committee.

This modeling approach is powerful but also introduces complexity:

  • Committee membership is split between:
    • Custom fields (who holds which role), and
    • Event registrations (who is in which committee event).
  • Documentation and automation must treat both layers consistently, or rosters can drift out of sync.

6. Design & Customization Implications

Understanding the data model helps us decide how far to push customization and where to anchor new features.

  • Good candidates for automation:
    • Leadership directories built from contact custom fields.
    • Committee pages driven by pseudo-event registrations.
    • API-based exports and dashboards keyed off membership level and roles.
  • Areas to use caution:
    • Designs that assume committees or roles are first-class WA objects (they are not).
    • Complex logic spread across both custom fields and pseudo-events without clear documentation.
    • Any change that might accidentally expose committee pseudo-events in public calendars or listings.

Recommendation: When proposing new features or automations, always specify which WA object(s) the feature depends on (contacts, custom fields, events, registrations, etc.). This keeps the mental model clear for future SBNC volunteers and developers.
