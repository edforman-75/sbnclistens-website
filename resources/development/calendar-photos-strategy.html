Objectives

SBNC wants a single calendar UI that serves both public visitors and members, plus a photo system that supports event-level organization, face recognition, member privacy, and easy downloads – all embedded cleanly within Wild Apricot and working across modern browsers and devices.

Assumptions about SBNC capabilities

  • We can use the Wild Apricot API to read and write data.
  • We can build integrations using Make.com or similar tools.
  • Therefore, solutions that require API-based sync or automation are acceptable and should not be disqualified on technical grounds.

Single Calendar Strategy – One UI for Public and Members

The guiding principle is: one source of truth, one calendar UI. Events are created and managed in Wild Apricot, while a modern calendar UI reads this data through the API and presents it in a filterable, mobile-friendly way.

Current Pain Points with Native WA Calendar

  • Buggy or inconsistent behavior in some browsers.
  • Limited, awkward filtering (tags/categories are under-exposed to end users).
  • Visually dated; not ideal as the public-facing “front door” calendar.
  • Poor mobile experience compared to modern calendars.

Primary Calendar UI: NewPath WaGadget CalendarPlus

CalendarPlus is built specifically for Wild Apricot and uses the WA API. It can act as the single calendar UI for both public visitors and logged-in members.

Key Features

  • Reads events directly from WA (no double entry).
  • Multiple views (list, tile, grid) with modern styling.
  • Filterable by tags, category, registration status, and date.
  • Color-coding by event type or other categories.
  • Embeddable on both public pages and member-only pages.
  • Respects WA event visibility when configured correctly.

How “One Calendar” Works in Practice

  1. All events are created in Wild Apricot with:
    • Visibility: Public vs Members-only / specific member levels.
    • Tags for activity type (Hikes, Wine, 20–40s, Social, Committee, etc.).
  2. CalendarPlus is configured to:
    • Respect WA visibility (public visitors only see public events; members see public + member-only events they’re allowed to view).
    • Expose filters based on SBNC’s tag taxonomy so users can drill into the events that matter to them.
  3. The same CalendarPlus configuration (or a cloned profile) is embedded:
    • On a public “Events” page (marketing view).
    • On a members-only “Member Events / Activities” page (full view).

Role of the Native WA Calendar

Given SBNC’s experience (bugs, limited features, aesthetics), the native WA calendar is not recommended as the primary UI. It can be kept as:

  • A hidden admin-only view for troubleshooting.
  • An emergency fallback if CalendarPlus fails.

Where Tockify Fits

Tockify provides an excellent standalone calendar with beautiful layouts. However, it does not natively read WA events. To use it seriously, we would either:

  • Double-enter events in WA and Tockify (not preferred), or
  • Use Make.com + WA API to sync events into Tockify (more moving parts).

Because SBNC can integrate systems, a WA→Tockify sync is technically feasible, but it introduces more operational risk than a pure WA+CalendarPlus setup. Tockify is best considered a “nice to have” design option if the committee strongly prefers its look.

Photo Strategy – Event Albums, Tagging, and Member Privacy

Requirements

  • Organize photos by event, activity type, and optionally by member.
  • Support face recognition and tagging (directly or via workflow tools).
  • Allow members to download photos (full or high-res copies).
  • Allow members to request removal or hiding of photos containing them.
  • Embed event galleries on WA pages (event recaps, highlight pages, “Why Join” content).

Option A – Wild Apricot Native Photo Albums

  • Pros:
    • Stored directly in WA; permissions match public vs member-only areas.
    • Simple: create an album gadget on a page and upload images.
    • No extra subscriptions or systems to manage.
  • Cons:
    • No face recognition or robust tagging by person.
    • Limited metadata: essentially captions only.
    • Not great for large or long-lived archives.
    • Clumsy to support “please remove all photos of me” requests.

Option B – SmugMug Embedded Galleries (Recommended)

  • Pros:
    • Designed for serious photo archives with folders, albums, and strong search.
    • Works well with Lightroom and other desktop workflows.
    • Supports keyword tagging and, with helper tools, face recognition.
    • Download controls (allow/disallow, original vs smaller sizes).
    • Album-level privacy controls (unlisted, password-protected, etc.).
    • Embeds cleanly into WA via iframe or script widgets.
    • Central location for implementing member removal requests: update once in SmugMug, and all embedded galleries reflect the change.
  • Cons:
    • Separate subscription and login.
    • Requires some upfront work to define folder/album structure and tagging conventions.

Option C – Flickr + POWR Gallery / Elfsight / Google Photos

These provide various mixes of embedding and gallery features, but they generally fall short vs a SmugMug-based approach on one or more of:

  • Consistent face recognition and tagging.
  • Strong privacy controls and member “hide me” workflows.
  • Long-term platform stability and support.

Recommended Photo Architecture

  1. Use SmugMug as the canonical photo archive for all SBNC event photos.
    • One album per event.
    • Tags for activity type (Hike, Wine, Social, etc.).
    • Optional tags or face recognition for individual members.
  2. Embed SmugMug albums in WA:
    • Each event recap page includes its matching SmugMug album.
    • Public highlight pages feature curated SmugMug albums or composite galleries.
  3. Handle member privacy:
    • Provide a simple “Request photo removal” form (WA or Jotform).
    • Admins hide/remove the relevant images in SmugMug; embedded galleries automatically reflect the change.
  4. Optional automation:
    • Use Make.com + WA API to keep a mapping between WA event IDs and SmugMug album IDs.
    • Optionally maintain an Airtable “photo index” (photo → event → member) for internal use.

Build vs. Buy – Calendars and Photos

Custom-Build Scenario

A fully custom calendar and photo system (React front-end, custom database, AWS Rekognition for faces, etc.) is feasible but would involve:

  • New hosting and application infrastructure.
  • Custom admin UI for events, photos, and tagging.
  • Ongoing security and maintenance responsibilities.
  • Significant initial cost and a non-trivial risk of abandonment if volunteers roll off.

This approach is likely overkill for SBNC’s needs and volunteer capacity.

Buy / Integrate Scenario (Recommended)

  • Calendars – Keep WA as the event database, use CalendarPlus as the single UI.
  • Photos – Use SmugMug as the archive; embed galleries into WA.
  • Automation – Use Make.com to glue things together where needed (e.g., mapping events to albums, syncing data for analytics).

This keeps complexity manageable while delivering a modern experience for members and prospects.

iframes in Wild Apricot – Usage, Reliability, and Mobile Behavior

Where iframes Typically Appear

  • Calendars
    • CalendarPlus usually uses the WA gadget framework and JavaScript embeds, not plain iframes, which tends to behave better responsively.
    • Tockify can be embedded via JS or iframe. The iframe method is common for simple use.
  • Photo Galleries
    • SmugMug albums are commonly embedded with an iframe or JS snippet that renders an iframe.
    • Flickr/POWR and Elfsight galleries often inject content via iframes.
  • Other Embeds
    • Airtable views, some forms, and some chat widgets rely on iframes.

Reliability & Security Considerations

  • HTTPS everywhere – All SBNC pages and external widgets must be loaded over HTTPS to avoid mixed-content warnings and blocking in modern browsers.
  • Vendor uptime – If a third-party service is down, the iframe area will appear blank or show an error, but the rest of the WA page will continue to work.
  • Cross-origin restrictions – For security, browsers prevent us from directly reading or modifying the internals of third-party iframes. Responsiveness and UX must be handled by the vendor’s code.

Mobile, Tablet, and Cross-Browser Behavior

  • Responsive width
    • Our CSS should ensure iframes sit in containers that are 100% width on small screens.
    • Most modern widgets are mobile-aware and adjust width automatically.
  • Height and scrolling
    • Nested scrollbars (page + internal iframe scroll) can be annoying on phones.
    • Prefer widgets that auto-size their height; if not possible, set a reasonable fixed height and accept a scroll within the iframe.
  • Privacy blockers
    • Safari’s Intelligent Tracking Prevention and similar features can block some tracking cookies, but for calendars/maps/galleries this usually does not break basic display.

iframes vs JavaScript Embeds

  • Preferred – JavaScript widgets that render content into a WA content gadget. These usually behave more like native page components and can be easier to style responsively.
  • Acceptable – iframes for SmugMug, Airtable, maps, and specialized widgets, as long as:
    • They are wrapped in responsive containers.
    • They are tested in Chrome, Safari, Firefox, Edge, and on iOS/Android devices.

Summary of Recommended Approach

  • Calendar – NewPath WaGadget CalendarPlus as the single, filterable calendar UI for both public and members; WA remains the event database.
  • Photos – SmugMug as the main photo archive, with embedded galleries on WA pages and a simple workflow for member removal requests.
  • Integrations – Use WA API and Make.com where appropriate (e.g., mapping events to albums, analytics, or automation), taking advantage of SBNC’s technical capability.
  • iframes – Use iframes and JS embeds confidently, but test each major widget across desktop browsers and on phones/tablets before going live.
